IP Addresses

IP Addresses are nothing but the locations of IP in network routing system.
It contains 4 bytes (aka. four octets), and was commonly written in "dots and numbers".
example = 178.42.72.3

There are also two types of IP addresses
1. IPv4 (internet protocol version 4)
2. IPv6 (internet protocol version 6)

the example we just show is a of IPv4

IPv6 : 2001:0db8:c9d2:aee5:73e3:934a:a5ae:9551
we can also write it for shorthand like 

2001:0db8:c9d2:0012:0000:0000:0000:0051
-> shorthand for 
-> 2001:db8:c9d2:12::51

you see we eliminate all the leading zeros

Subnets

the wikipedia defination of subnet is -
A subnet is a logical subdivision of an IP network. The practice of dividing the network
into two or more networks is called subnetting.

for instance in IPv4 192.0.2.12 we could say that first 3 bytes are the network 
    and last byte was the host. That means there is a host 12 on network 192.0.2.0

htonl()	    host to network long

these are some functions used to convert the bytes as needed

Struct's

socket is of type `int`
there is a struct name addrinfo - 

struct addrinfo {
    int              ai_flags;     // AI_PASSIVE, AI_CANONNAME, etc.
    int              ai_family;    // AF_INET, AF_INET6, AF_UNSPEC
    int              ai_socktype;  // SOCK_STREAM, SOCK_DGRAM
    int              ai_protocol;  // use 0 for "any"
    size_t           ai_addrlen;   // size of ai_addr in bytes
    struct sockaddr *ai_addr;      // struct sockaddr_in or _in6
    char            *ai_canonname; // full canonical hostname

    struct addrinfo *ai_next;      // linked list, next node
};

here ai prefix is addrinfo

then there is sockaddr (socket address) struct 

struct sockaddr {
    unsigned short sa_family; // address family, AF_xxx
    char           sa_data[14]; // 14 bytes of protocol address
}

struct sockaddr is protocol agonostic, this is a generic structure used by socket 
apis so they can handle any type of address (ipv4, ipv6, unix domain, etc)

- Acts like a base/abstract type
- Used in function signatures like bind(), connect(), accept()
- You usually don’t fill this directly
- sa_data is kind of a blob — not convenient to use

focus on 3rd point, we do not want to pack address in the sa_data, that could change the byte order or 
undefined behavior, or who knows what. To tackel this programmer came with

struct sockaddr_in! in here is refers to internet

struct sockaddr_in {
    short int          sin_family;  // Address family, AF_INET
    unsigned short int sin_port;    // Port number
    struct in_addr     sin_addr;    // Internet address
    unsigned char      sin_zero[8]; // Same size as struct sockaddr

}

this is specialized structure for ipv4 addresses.

- Designed specifically for IPv4
- Fields are meaningful and easy to use:
    - sin_port → port number
    - sin_addr → IP address
- Much more user-friendly

now there is struct sockaddr_storage

struct sockaddr_storage {
    sa_family_t  ss_family;     // address family

    // all this is padding, implementation specific, ignore it:
    char      __ss_pad1[_SS_PAD1SIZE];
    int64_t   __ss_align;
    char      __ss_pad2[_SS_PAD2SIZE];
}

This can be used if you don't know what type of address you are dealing with in compile time.
Meaning it has a capability to store ipv4 and ipv6 both of the address
